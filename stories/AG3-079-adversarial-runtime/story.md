## 3. Akzeptanzkriterien

0. **Mandatory-Target-Ableitung FK-48-§48.2.2-konform:** `extract_mandatory_targets` leitet Targets aus Layer-2-Findings vom Typ `assertion_weakness` (Status `FAIL`/`PASS_WITH_CONCERNS`) ab, nicht pauschal pro `Severity.BLOCKING`; jedes Target traegt `addressed_part`; ein BLOCKING-Finding **ohne** `assertion_weakness`-Typ erzeugt **kein** mandatory target (zwei Tests: assertion_weakness -> Target mit `addressed_part`; reines BLOCKING ohne Typ -> kein Target). Die §48.2.3-Prompt-Sektion "Mandatory Targets" wird nur bei vorhandenen Targets eingefuegt (Test).
1. `AdversarialChallenger.evaluate` ist kein Passthrough mehr: ein Lauf ohne ausgefuehrten Test fuehrt zu Layer-3 FAIL, nicht zu stillem PASS (Test).
2. Mindestens ein Test wird ausgefuehrt; Tests werden ausschliesslich in die Sandbox (`_temp/adversarial/{story_id}/{epoch}/`) geschrieben, Produktivcode bleibt unveraendert (Test an der Phasengrenze).
3. Pflicht-Sparring durchgesetzt: fehlt **eines** der beiden Sparring-Telemetrie-Fakten (`EventType.ADVERSARIAL_SPARRING` mit `pool`-Feld **oder** `EventType.LLM_CALL` mit `role=adversarial_sparring`), schlaegt der Pflichtnachweis fail-closed fehl (Tests fuer beide Faelle).
4. **Test-Promotion, drei deterministische Pfade** ueber **in der Sandbox erzeugte** Tests (FK-48 §48.1.5):
   - Sandbox-Test schema-valide **und** im Trockenlauf ausfuehrbar **und** nicht-dupliziert **und** Test PASS -> nach `tests/`;
   - schema-valide/ausfuehrbar/nicht-dupliziert **und** Test FAIL (= nachgewiesener Befund) -> nach `tests/adversarial_quarantine/`;
   - schema-invalide **oder** Trockenlauf-Fehler **oder** Duplikat -> bleibt ephemer in der Sandbox, keine Promotion.
   Dedup-Kriterium benannt: ein Sandbox-Test gilt als Duplikat, wenn ein Test mit identischem Modul-qualifizierten Testnamen bereits unter `tests/` existiert. Je ein Test pro Pfad.
5. `adversarial.json` wird von einem Pipeline-Skript aus dem Sandbox-`result.json` materialisiert: Producer == `ADVERSARIAL_PRODUCER` (`verify-system.layer-3-adversarial`), Stage == `ADVERSARIAL_STAGE` (`qa-layer-adversarial`), `schema_version` == `"3.1"` mit `mandatory_target_results`; Sub-Agents koennen nicht in `_temp/qa/` schreiben (Test fuer Schutz). Producer/Stage werden aus den `qa_artifact_names`-Konstanten bezogen, nicht als Literal (Test pinnt die Werte).
6. Alle fuenf Adversarial-Telemetrie-Events werden mit den FK-48-§48.1.8-Erwartungswerten emittiert (genau 1 `start`/`end`, >= 1 `sparring`/`test_executed`, >= 0 `test_created`) (Test).
7. Das Integrity-Gate prueft bei Closure den Sparring-Pflichtnachweis (>= 1 `adversarial_sparring` **und** >= 1 `llm_call role=adversarial_sparring`) sowie die §48.1.8-Erwartungswerte zusaetzlich zum bestehenden Envelope-Nachweis (Dim 6); fehlende Events -> Gate FAIL (Test). Der bestehende `NO_ADVERSARIAL`-Envelope-Check bleibt unveraendert gruen fuer einen konformen Lauf.
8. Gate-Rueckkopplung: ein nicht erfuelltes mandatory target setzt das ueber `target_id -> (layer, check)` gemappte Layer-2-Finding auf mind. `FindingResolutionStatus.PARTIALLY_RESOLVED` fuer die naechste Runde; ein erfuelltes (`TESTED`+Test-PASS oder begruendetes `UNRESOLVABLE`) nicht (Tests beider Faelle); geschrieben ins bestehende Resolution-/RemediationFeedback-Modell, kein neuer Status-Automat.
9. **Pflichtbefehle gruen:** pytest unit/integration/contract (in Chunks, `-n0`); mypy default + `--platform linux`; ruff; vier Konzept-Gates; Coverage >= 85 %.
