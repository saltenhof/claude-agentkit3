# Codex Review R1 - AG3-140

## 1. Summary

REJECT. The implementation removes the obvious `op_id` server defaults and the new generic guard has the right core shape for `story_context_manager` / `task_management`, but the delivered scope is not complete. At least one mutating route still has no `op_id`, multiple mutating BC routes only echo `op_id` without replay/body-hash/in-flight protection, and the control-plane/project-edge paths do not satisfy the same-body/different-body contract. The route inventory is also materially inaccurate.

I did not modify source or test files. I did not run the test suite; this review is based on the requested diff and full-context source inspection.

## 2. Findings

1. `src/agentkit/backend/execution_planning/http/routes.py:228` - BLOCKER - `DELETE /v1/projects/{project_key}/planning/dependencies/{story_id}/{depends_on_story_id}/{kind}` is still a mutating route with no `op_id` at all. The handler removes the dependency at `src/agentkit/backend/execution_planning/http/routes.py:249` and returns a committed mutation response at `src/agentkit/backend/execution_planning/http/routes.py:263`, but `ControlPlaneApplication` dispatches DELETE bodies only to auth before calling planning, so this route cannot fail closed with 422 on missing `op_id`, cannot hash the URL target, cannot replay the stored result, and cannot reject a parallel duplicate. Failure scenario: a client times out after the dependency is removed, retries the same DELETE, and gets `story_dependency_not_found` rather than the stored success; a parallel duplicate can race through the same unguarded mutation path. Suggested fix: add an explicit idempotency contract for dependency removal, including a required client `op_id` carried in the DELETE body or a deliberate route redesign, hash the path tuple, claim before `remove_dependency`, finalize deterministic 2xx/4xx, add missing-op_id/body-mismatch/replay/in-flight tests, and add it to the inventory.

2. `src/agentkit/backend/project_management/http/routes.py:501`, `src/agentkit/backend/execution_planning/http/routes.py:273`, `src/agentkit/backend/auth/http/routes.py:212` - MAJOR - Several mutating BC routes were changed only to require `op_id`; they do not implement the AG3-140 contract. Project create/archive/patch return `_mutation_response` with the supplied `op_id` but never claim/finalize (`project_management/http/routes.py:563`, `:607`, `:346`, `:453`). Execution dependency create/config upsert similarly just echo `op_id` (`execution_planning/http/routes.py:321`, `:497`). Auth token create/revoke do the same (`auth/http/routes.py:230`, `:257`). Failure scenario: two identical `POST /v1/projects/{key}/api-tokens` calls with the same `op_id` mint two different plaintext tokens instead of replaying the first response; a reused `op_id` with a different token label is not `409 idempotency_mismatch`; a parallel duplicate is not fenced. Suggested fix: put these routes under the unified guard or a BC-owned equivalent on `control_plane_operations`, with canonical body hashes including URL path keys, stored response replay, mismatch 409, in-flight rejection/serialization, and route-level negative tests.

3. `src/agentkit/backend/control_plane/runtime.py:377` - MAJOR - The control-plane phase/closure replay path still replays terminal rows by `op_id` alone and never compares request body/operation shape. `_build_claim_placeholder` constructs claimed records without `request_body_hash` at `src/agentkit/backend/control_plane/runtime.py:2613`, `_operation_record` also omits it at `src/agentkit/backend/control_plane/runtime.py:2493`, and `_claim_operation` returns `_replayed_result(stored.response_payload)` for any terminal status at `src/agentkit/backend/control_plane/runtime.py:381`. Failure scenario: a client successfully starts `setup` with `op_id=op-x`, then accidentally uses `op-x` for a different phase/action/body; the loser receives the stored setup result instead of `409 idempotency_mismatch`. Suggested fix: compute a canonical hash for every phase/closure request, stamp it on claim/finalize, and classify terminal claim losers as replay vs mismatch before returning a stored result.

4. `src/agentkit/backend/control_plane/runtime.py:2071` - MAJOR - `POST /v1/project-edge/sync` is not persisted as an idempotent operation at all. The HTTP handler calls `sync_project_edge` at `src/agentkit/backend/control_plane_http/app.py:1487` and returns the freshly computed result at `src/agentkit/backend/control_plane_http/app.py:1513`; the runtime method builds a `ControlPlaneMutationResult` but never claims, stores, or replays a `control_plane_operations` row. Failure scenario: retrying the same `op_id` after state changes returns a different edge bundle instead of the stored result; a parallel duplicate is not fenced. Suggested fix: wrap project-edge sync in the same claim/body-hash/finalize path or document and remove it from the mutating idempotency inventory if it is truly a read-only observation.

5. `src/agentkit/backend/story_context_manager/service.py:1139` - MAJOR - `story_context_manager` releases claims on every exception after a fresh claim, including deterministic domain failures, so replay-after-failure is not stored. The status transition path claims, then on `InvalidStatusTransitionError`, `ForbiddenError`, etc. releases at `src/agentkit/backend/story_context_manager/service.py:1156` instead of finalizing a deterministic error result; create/update/cancel have the same pattern at `:509`, `:602`, `:745`, `:939`, and `:1058`. Failure scenario: `POST /v1/stories/{id}/approve` with a valid `op_id` against a non-Backlog story returns a deterministic 422, releases the row, and a retry re-enters the service rather than replaying the stored failure, contrary to AC8's replay-after-failure requirement. Suggested fix: distinguish deterministic 4xx/domain outcomes from unexpected 5xx/infrastructure failures, finalize the former into the idempotency record, and only release claims for failures that happened before a committed/deterministic outcome.

6. `stories/AG3-140-unified-idempotency-contract/route-inventory.md:59` - MAJOR - The mandatory route inventory is not accurate or complete. It lists execution config as `.../execution-input/limits`, but the implemented mutating route is `/v1/projects/{project_key}/planning/config` (`src/agentkit/backend/execution_planning/http/routes.py:64`). It omits the mutating story field PUT wired through `ControlPlaneApplication` (`src/agentkit/backend/control_plane_http/app.py:1252`, `src/agentkit/backend/story_context_manager/http/routes.py:251`). It omits the mutating dependency DELETE above. It lists auth token routes as `/tokens` at `route-inventory.md:68`, while the actual routes are `/api-tokens` (`src/agentkit/backend/auth/http/routes.py:28`). It also claims project-edge sync has claim/finalize coverage even though the runtime does not persist the operation. Suggested fix: regenerate the inventory from the actual route regexes/dispatch table, add exact mechanism and evidence per route, and make the evidence tests prove the stated contract rather than just op_id validation.

## 3. Assessment of Worker WARNINGs

(a) Generic-guard orphan claims with `backend_instance_id = NULL`: acceptable deferral only if explicitly tracked to AG3-138/AG3-142. The behavior is fail-closed because retries see `operation_in_flight` until admin abort; it does not create double execution. It is operationally painful, but not an AG3-140 rejection by itself.

(b) Guard-counter `correlation_id` inert: acceptable as a short-term compatibility wart, not a contract defect. It should either be removed from the repository signature once callers can change, or documented as intentionally unused at that boundary.

(c) Task-management control_plane_http integration using the in-memory guard: acceptable for the SQLite/docker-free allow-list only if it is not presented as real-store route coverage. The low-level Postgres guard tests cover the record semantics, but the missing full-contract coverage on other routes remains a separate issue.

## 4. Route-Inventory Coverage Check

The inventory is not a reliable completeness artifact. It correctly identifies the five task-management POST routes and the worker-health exception, but it misses active mutating routes, uses wrong paths for auth/config, and overstates mechanisms for project-edge sync and non-guarded BCs. It also cites broad test files for project/auth/execution routes where the current tests do not prove replay, body mismatch, in-flight rejection, or replay-after-failure. AC5 is therefore not met.

VERDICT: REJECT
