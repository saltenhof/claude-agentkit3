## 2. Scope

### 2.1 In Scope

1. **`PatternPromotion`-Sub** (FK-41 ┬¦41.5): `suggest_patterns()` real ŌĆö deterministisches Clustering der `OBSERVED`-Incidents + Promotion-Regel-Auswertung als typisierte Regeln (`PromotionRule`-StrEnum, **englische Werte**). Output: `PatternCandidate`-Liste mit Incident-Refs, Invariant-Kandidat, Promotion-Regel. **Deterministische Cluster-Regel (testbar, ersetzt das unbestimmte "Symptom-Aehnlichkeit"):**
   - Cluster-Schluessel = `(FailureCategory, symptom_signature)`.
   - **`symptom_signature` ŌĆö vollstaendig spezifizierte, deterministische Normalisierung des `symptom`-Freitexts (kein LLM, keine Fuzzy-Schwelle):**
     1. Unicode-NFKC-Normalisierung, dann ASCII-Faltung (nicht-ASCII auf nahesten ASCII abbilden, sonst verwerfen).
     2. Lowercase.
     3. Tokenisierung an der Regex-Klasse `[^a-z0-9]+` (alle Nicht-Wort-Zeichen als Trenner; Ziffern bleiben erhalten).
     4. Leere Tokens verwerfen. **Keine Stopword-Entfernung** ŌĆö es gibt bewusst keine Stopword-Liste; jedes nicht-leere Token zaehlt (deterministisch, sprachneutral; eine Stopword-Liste waere eine nicht-normierte Heuristik und ist ausgeschlossen).
     5. Tokens lexikografisch sortieren, mit Einzel-Leerzeichen joinen.
     6. `sha256` der UTF-8-Bytes des Join-Strings; `symptom_signature` = die ersten 16 Hex-Zeichen (8 Bytes) des Hexdigests.
   - Tie-Breaker bei gleichem Cluster-Schluessel: aelteste `recorded_at` zuerst (stabile Reihenfolge).
   - **Promotion-Regeln** je Cluster (FK-41 ┬¦41.5.1), genau einer der drei Werte wird gesetzt (Prioritaet `HIGH_SEVERITY` > `REPETITION` > `FAVORABLE_CHECKABILITY`):
     - `REPETITION`: `>= 3` Incidents gleichen Cluster-Schluessels mit `recorded_at` innerhalb 30 Tagen (Fenster aus Config, siehe ┬¦2.2).
     - `HIGH_SEVERITY`: `>= 1` Incident mit `severity in {IncidentSeverity.HIGH, IncidentSeverity.CRITICAL}` (= "produktionsrelevanter oder sicherheitskritischer Impact", FK-41 ┬¦41.5.1, deterministisch ueber die englische `IncidentSeverity`-Skala `failure_corpus/types.py:47-50`).
     - `FAVORABLE_CHECKABILITY`: `>= 2` Incidents UND der **deterministisch der Kategorie zugeordnete Check-Typ** (ueber die im Konzept normierte **FK-41-┬¦41.6.3-KategorieŌåÆCheck-Typ-Matrix**, dieselbe Konstante wie in Schritt 2, ┬¦2.1.3) traegt False-Positive-Risiko `LOW`. Das FP-Risiko je Check-Typ ist eine **benannte, getestete, story-eigene Konstante `CHECK_TYPE_FALSE_POSITIVE_RISK`** mit exakt diesen Werten (deterministisch, kein LLM, kein Human-Review vor der Promotion; die Werte spiegeln die Determinierbarkeit/Eindeutigkeit des jeweiligen Check-Typs):

       | `CheckType` (FK-41 ┬¦41.6.3) | `FalsePositiveRisk` |
       |---|---|
       | `CHANGED_FILE_POLICY` | `LOW` |
       | `SENSITIVE_PATH_GUARD` | `LOW` |
       | `FORBIDDEN_DEPENDENCY` | `LOW` |
       | `ARTIFACT_COMPLETENESS` | `MEDIUM` |
       | `TEST_OBLIGATION` | `MEDIUM` |
       | `FIXTURE_REPLAY` | `HIGH` |

       Damit ist `FAVORABLE_CHECKABILITY` genau dann erfuellt, wenn `len(incidents) >= 2` UND `CHECK_TYPE_FALSE_POSITIVE_RISK[check_type_for_category(category)] == FalsePositiveRisk.LOW`. Ein Test pinnt die vollstaendige Matrix und die abgeleitete Pro-Kategorie-Checkbarkeit.
   - Cluster, die keine der drei Regeln erfuellen, erzeugen **keinen** Kandidaten.

Bei `REJECTED`: Status `REJECTED`, kein aktives Pattern. "Pattern ohne Check ist reines Lese-Artefakt" (┬¦41.5.5): kein Guard, keine Pipeline-Wirkung.
