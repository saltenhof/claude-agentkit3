## 2. Scope

### 2.1 In Scope
1. **`PlannedStory` als typisiertes Planungs-Read-Model (§70.4.1):** `StoryRefForPlanning` wird auf den §70.4.1-Pflichtfeldsatz gehoben (bzw. ein klar benanntes `PlannedStory`-Read-Model bereitgestellt) mit `project_key`, `story_id`, `story_type`, `story_size`, `participating_repos: tuple[str, ...]`, `human_touchpoints`, `external_prerequisites`, `planning_status: PlanningStatus`. Diese Felder sind reine in-memory Read-Model-Felder; die Befuellung aus der autoritativen Story-Persistenz ist AG3-099 (Ingest/Persistenz) bzw. AG3-100 (Surface). Bestehende Aufrufer von `StoryRefForPlanning` bleiben kompatibel (neue Felder mit fail-closed Defaults, kein Bruch von AG3-054).
2. **`BlockingCondition` als typisiertes Objekt (§70.4.3):** geschlossenes StrEnum der sechs Blockerklassen + ein Modell, das Blocker auswertbar an eine Story bindet (mit Provenienz/Grund-Referenz, nicht nur Freitext). `WaveStory.blocked_by` wird auf typisierte Blocker gehoben (kein roher String-Listen-Truth mehr fuer die Blocker-Semantik; die Story-ID-Liste bleibt fuer `blocked_internal_dependency` ableitbar).
3. **`PlanningStatus` als StrEnum (§70.5.1):** genau die acht §70.5.1-Werte `UNSTARTED`, `READY`, `FLIGHT`, `DONE`, `BLOCKED_EXTERNAL`, `BLOCKED_HUMAN`, `BLOCKED_CAPACITY`, `BLOCKED_CONFLICT`. Orthogonal zu BC-3-`StoryStatus` (typisierte, abgeleitete Achse). **Verbindliche Blockerklasse→PlanningStatus-Abbildung** (deterministische Funktion, da FK sechs Blockerklassen aber nur vier `BLOCKED_*`-Status kennt — Abbildung statt 1:1-Spiegelung):

   | `BlockingCondition`-Klasse | `PlanningStatus` |
   |---|---|
   | `blocked_internal_dependency` | `UNSTARTED` (harter Vorgaenger nicht DONE; reiner Graph-Zustand, kein eigener BLOCKED_*-Status — §70.6.1 #1) |
   | `blocked_external` | `BLOCKED_EXTERNAL` |
   | `blocked_human` | `BLOCKED_HUMAN` |
   | `blocked_capacity` | `BLOCKED_CAPACITY` |
   | `blocked_conflict` | `BLOCKED_CONFLICT` |
   | `blocked_contract` | `BLOCKED_CONFLICT` (geschuetzte Vertragsflaeche ist ein Konfliktzustand i.S.v. §70.6.1 #5) |

   Liegen mehrere Blocker an, gilt eine deterministische Prioritaet (`BLOCKED_HUMAN` > `BLOCKED_EXTERNAL` > `BLOCKED_CONFLICT` > `BLOCKED_CAPACITY` > `UNSTARTED`-aus-internal), damit die Status-Ableitung eindeutig und reproduzierbar ist.
4. **Feasibility-vs-Scheduling-Trennung (§70.4.4/§70.6.2):** zwei getrennte, typisierte Ergebnisse — `ExecutionFeasibility` (`can_parallelize`, aus harten Graph/Gate/Konflikt-Regeln) und `ExecutionSchedulingPolicy` (`may_parallelize_now`, aus Budgets). Budget-Dimensionen als typisiertes Modell: `repo_parallel_cap`, `merge_risk_cap`, `api_rate_limit_cap`, `llm_pool_cap`, `ci_capacity_cap`. Die normative Auswertungsreihenfolge (harte Regeln -> zentrale Budgets `max_allowed_batch` -> projektlokale Hints verengen `recommended_batch`, heilen nie eine Feasibility-Verletzung) ist als deterministische Funktion abgebildet. `compute_readiness` wird so umgebaut, dass es die beiden Ebenen NICHT mehr in ein bool/`practical_parallelism` kollabiert (`readiness.py:44-47`).
5. **Typisierte Kanten in der Feasibility-Schicht (§70.4.2/§70.6.1):** die Graph-/Feasibility-Schicht konsumiert typisierte Kanten (`StoryDependencyKind`) und wertet nur **harte** Kanten (`hard_story_dependency`, `serial_execution_constraint`, `mutex_constraint`, `shared_contract_dependency`, `shared_file_conflict`, `external_dependency`, `human_gate_dependency` gem. §70.6.1) blockierend; `soft_story_dependency` blockiert die Feasibility NICHT. Die bestehende kindlose `DependencyGraph`-Topologie (`dependency_graph.py:17-25`) wird so erweitert oder ergaenzt, dass das Kind verfuegbar bleibt; die reine `has_cycle`-Erkennung bleibt unveraendert.
6. **`SchedulingHint` als nicht-persistentes Eingabe-Value-Type (§70.6.2):** ein typisiertes, in-memory `SchedulingHint`-Modell, das `recommended_batch` **verengen** darf (nie freigeben). Es ist KEIN compiliertes Rulebook und KEINE Persistenz — der Rulebook-Compile/`rulebook_revision`-Schreibpfad ist AG3-099. Diese Story stellt nur den deterministisch auswertbaren Hint-Input bereit, den die Auswertungsreihenfolge (Scope 4) konsumiert.
7. **`HumanGate` / `ExternalGate` als erstklassige Planungsobjekte (§70.5.3/§70.5.4):** typisierte Gate-Modelle; nur ein blockierendes Human-Gate erzeugt `BLOCKED_HUMAN`, optionale Human-Review ist nicht-blockierend (§70.6.1 letzter Satz). Gate-Resolved-Zustand modelliert.
8. **PlanDerivation-Outputs (§70.6.4):** typisierte Ableitung `critical_path`, `ready_set`, `blocked_set`, `execution_wave`, `recommended_batch`, `max_allowed_batch` aus dem Graphen + Feasibility + Scheduling. Owner der Berechnung ist diese reine Ableitung im BC; das Story-Read-Model-Feld `critical_path` (`story_model.py:207`) bleibt eine nachgelagerte Projektion (keine zweite Wahrheit). Readiness-Regel exakt nach §70.6.1.
9. **`ExecutionWave` mit Lifecycle (§70.4/§70.6.4):** typisiertes Wave-Objekt mit Lifecycle-StrEnum `planned`/`active`/`completed`/`collapsed`; teilweises Scheitern markiert die Wave als `collapsed` (kein stiller Inkonsistenz-Zustand). Tenant-scoped (`project_key`), nie projektuebergreifend aggregiert.
10. **`RePlanTrigger` als typisiertes Klassifikations-Modell (§70.6.2a):** geschlossenes StrEnum der fuenf Trigger-Klassen (`story_done`, `blocker_or_gate_changed`, `capacity_budget_changed`, `rulebook_or_policy_changed`, `conflict_or_contract_reevaluated`) + eine reine Funktion, die eine Aenderungsbeschreibung deterministisch auf die Trigger-Klasse abbildet. **Nur das typisierte Vertragsmodell + reine Klassifikation** — die laufzeit-/eventgetriebene, debounced Durchsetzung (Emission, Revision, Anti-Thrashing) ist AG3-099 (Events/Persistenz) bzw. AG3-100 (Pflichtaufruf im Lauf).
11. **Reine Ableitungslogik + Tests:** alle Ableitungen als deterministische reine Funktionen (gleiche Eingabe -> gleiche Ausgabe). Tests: §70.4.1-Pflichtfeldsatz vorhanden; jede Blockerklasse erzeugt den korrekten PlanningStatus gem. Abbildungstabelle (Scope 3); Feasibility und Scheduling sind getrennt auswertbar; Readiness-Regel-Matrix (§70.6.1); `soft_story_dependency` beeinflusst Feasibility NICHT; typisierte Kante wird im Graph nicht verworfen; Wave-`collapsed` bei Teil-Scheitern; `SchedulingHint` kann nur verengen; jede `RePlanTrigger`-Klasse wird korrekt klassifiziert; Determinismus-Test.
